## 已确认的架构级变更 (2026-06-06)

### 加密架构：主密码 + 恢复密钥 双入口

**旧方案**：主密码 → PBKDF2 → AES 密钥，只有一条解密路径

**新方案**：

```
用户自定义密钥 (≥8位)          主密码
      │                          │
      ▼                          ▼
PBKDF2(keySalt,10万轮)    PBKDF2(mpSalt,60万轮)
      │                          │
      ▼                          ▼
    REK (32B)    ←───────  KEK 加密/解密 REK
      │
      ▼
AES-256-GCM 加密/解密 .pvault 文件
```

**两条解密路径**：
- 路径 A（日常）：主密码 → KEK → 解密 encryptedREK → REK → 解密密码库
- 路径 B（恢复）：密钥 → PBKDF2 → REK → 解密密码库（验证 SHA-256(REK) 哈希）

**文件头部（全部明文，不影响安全）**：

| 字段 | 用途 | 明文？ |
|------|------|--------|
| `version` | 格式版本 (2) | ✅ |
| `masterPasswordSalt` | 主密码 PBKDF2 salt | ✅ |
| `keySalt` | 密钥 PBKDF2 salt | ✅ |
| `encryptedRek` | KEK 加密的 REK | 密文 |
| `rekIv` | REK 加密 IV | ✅ |
| `rekAuthTag` | REK 加密认证标签 | ✅ |
| `rekHash` | SHA-256(REK)，验证密钥输入 | ✅ |
| `payloadIv` | 密码库 IV | ✅ |
| `payloadAuthTag` | 密码库认证标签 | ✅ |

### 密钥规则

- 支持**系统随机生成**（32 字节 hex → 64 位字符）或**用户自定义**（≥8 位）
- 自定义密钥走 PBKDF2(10万轮) 拉伸为标准 32 字节
- 首次生成时：**全量展示一次**，带复制按钮
- 之后所有场景：**`a1b2****x7y8`** 格式（前 4 + 后 4，中间 4 个星号）
- **终生只能看到完整密钥一次**

### 四个场景

| 场景 | 流程 |
|------|------|
| 正常使用 | 锁屏 → 主密码 → 进入主界面 |
| 忘记密钥 | 主密码解锁 → 设置 → 导出明文 → 重新生成密钥 → 导入 |
| 忘记主密码 | 锁屏 → 重置所有数据 → 设新主密码 → 用旧密钥加载 .pvault（密钥不变） |
| 两个都丢 | **死档，无法恢复** |

### 多 Vault 数据模型（标签模式）

**核心改动**：从"Vault 包含 Entry"改为"Entry 标签化"。一个条目可以同时属于多个 Vault。

```
.pvault 加密 payload:
{
  vaults: [
    { id: 1, name: "默认", idPrefix: 10000, nextId: 10001 },   // 自动创建
    { id: 2, name: "工作", idPrefix: 20000, nextId: 20001 },
    { id: 3, name: "个人", idPrefix: 30000, nextId: 30001 }
  ],
  entries: [
    {
      id: 10001,           // 全局唯一，前缀标识所属 Vault
      website: "github",
      alias: "GitHub",
      account: "user@x",
      password: "enc...",
      description: "",     // 新增：备注
      vaultIds: [1, 2, 3], // 所属 Vault 列表
      visible: true,
      order: 0,
      createdAt, updatedAt
    }
  ]
}
```

**ID 规则**：每个 Vault 有自己的 ID 区间（10000-19999 / 20000-29999...），新建条目使用当前选中 Vault 的 nextId。`vaultIds` 独立管理多归属。
